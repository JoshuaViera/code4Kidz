System Architecture Overview

The full stack from a student's browser to the database. v3.0 adds an Observability layer — you now know when something breaks before a teacher files a complaint. Every layer has a single job. Every dependency is explicit.

v3.0 Upgrades in This View

The Event Bus layer now uses Redis Streams with Consumer Groups instead of Redis Pub/Sub — events are durable, replayable, and acknowledged. A new Observability layer (Datadog APM or Grafana Cloud) sits alongside the data layer. The compliance row now includes a DLQ (Dead Letter Queue) for failed analytics events.

Still Solid

Modular Monolith

8 domain modules, clear boundaries, extractable to microservices when needed. The right call for a startup — no distributed systems tax until you need scale.

v3.0 Upgrade

Redis Streams replaces Pub/Sub

Events are now durable. Consumer groups ensure each analytics event is processed exactly once. A DLQ catches failures after 3 retries and fires a P1 alert.

v3.0 New

Observability Layer

SLOs are defined per endpoint class. Error budgets are tracked. APM traces every request. You know when things break before a teacher files a ticket.

Organization Model & Role Hierarchy

Four organization types, six roles. Every action is scoped to an org boundary enforced at the database level via Row Level Security. No role can ever read data outside its tenant.

v3.0 Upgrade — Organization Timezone

Every org now carries a timezone field (defaults America/New_York). Streak-day boundaries, lesson-date calculations, and DoE report generation all run in org-local time — not UTC. This eliminates the 11:58pm streak-loss bug that plagues every edtech platform that doesn't think about this.

Onboarding Flows

Two parallel flows: adult account creation (invite-only, verified email) and anonymous student session creation (no email, no name, no PII). They never intersect.

Adult Onboarding (Instructor / Admin)

01
Invite Sent
Admin generates invite. invite_tokens row created. Email sent via Resend with signed URL. Token expires in 72h.
02
Accept & Register
Token validated (hash match, not expired, not used). User sets password. bcrypt 12 rounds. used_at stamped.
03
DPA Gate
If org's dpa_signed_at is null, user lands on a read-only DPA pending screen. No classroom access until DPA signed.
04
Org Active
Instructor can create classrooms. Admin can invite instructors. JWT issued with org_id + role claims.
05
Audit Logged
Every login, classroom creation, and roster action writes to audit_log. IP stored as SHA-256 hash only.

Student Session Creation (Anonymous)

01
Enter Join Code
Student types 6-char classroom code. No account required. No email. No age gate. Just a code.
02
Code Validated
Server confirms code exists and classroom is active. Returns org timezone for client-side streak display.
03
Slug Generated
3-word adjective-animal-color slug created. Profanity filtered. Stored in plaintext — it's a resume key, not a secret.
04
Session Stored
Row in student_sessions. Slug shown to student: "Remember your animal name to continue later." No password, no email.
05
Lesson Begins
Student starts Lesson 1. XP, streaks, and hearts tracked via session slug. Nothing else stored about this person.
Resume Flow

Student types their slug on any device → server looks up slug_hash → returns session state → lesson continues from last lesson_progress row. No account recovery, no "forgot password." It's a game save code.

Database Schema

PostgreSQL with Row Level Security enforced on every table that touches student or org data. v3.0 adds timezone to organizations, TIMESTAMPTZ to streak tracking, and bootstrap_median_ms for cold-start safety. v3.1 expands the mistake_log.error_type enum with two new values for CSS validation, and documents which new step types are excluded from DB writes entirely.

v3.0 Schema Changes — 3 Tables Updated

organizations: Added timezone TEXT NOT NULL DEFAULT 'America/New_York'. Source of truth for all streak day boundary calculations.
student_sessions: last_lesson_date changed from DATE to TIMESTAMPTZ. Streak logic uses date_trunc('day', last_lesson_date AT TIME ZONE org.timezone).
analytics_daily: Added bootstrap_median_ms INT. Seeded from lesson definition. Used as tier-3 fallback when live median has fewer than 10 samples.

v3.1 Schema Changes — 1 Table Updated + DB Write Exclusion Rules

mistake_log.error_type enum — 2 new values added:
css_syntax_error — student wrote a CSS rule missing a semicolon or colon. Fired by the css_has_semicolon validation rule introduced in Lesson 2.
css_property_error — student left a CSS property value empty or used an invalid value. Fired by the css_property_set validation rule.

DB write exclusion rules — NEW (affects lesson_progress + mistake_log):
warmup step type: writes NEITHER to lesson_progress NOR mistake_log. Auto-advances after time_limit_ms. The server ignores any progress POST from a warmup step — idempotency key is never generated.
anatomy_builder step type: writes NEITHER to lesson_progress NOR mistake_log. Wrong drag-and-drop attempts are client-side only. No server round-trip on incorrect drops. XP is always 0. The server only receives a completion signal when all labels are correctly placed — one POST, no retries needed.

analytics_daily — warmup + anatomy_builder exclusion: Steps of type warmup or anatomy_builder must NOT be included in analytics_daily aggregates for median_time_ms or top_error_type. These steps are orientation, not evaluation. Including them would skew the traffic light thresholds. Filter on step type before XADD to the analytics stream.

v3.0 — Timezone-Safe Streak Calculation

The bug: storing last_lesson_date as DATE without timezone context means a student in EST doing a lesson at 11:58 PM, then at 12:02 AM, loses their streak — the server sees two different dates because it's computing in UTC. The fix: store TIMESTAMPTZ, look up org timezone, and use date_trunc('day', ts AT TIME ZONE org.timezone) for all streak comparisons. The org's timezone is the canonical clock for that classroom.

Data Pipeline — From Student Action to Insight

v3.0 replaces the fire-and-forget Redis Pub/Sub event bus with Redis Streams + consumer groups. Events are now durable, acknowledged, and replayable. The lesson efficacy formula is Laplace-smoothed to eliminate division-by-zero and sample-size distortion. Traffic light thresholds have a 3-tier cold-start fallback so day-zero classrooms always have a valid baseline.

The Critical Fix — Redis Pub/Sub → Redis Streams

Redis Pub/Sub is fire-and-forget. If the analytics consumer is down when a student completes a step, that event is gone. Your analytics_daily table silently undercounts. Your efficacy scores drift. Your traffic light thresholds become wrong — without any indication that this happened. Redis Streams with a consumer group gives you durable delivery, acknowledgment, and a dead-letter queue for failed events. Same Redis instance. Minimal API change. Zero excuse not to do it.

v3.0 Fix — Lesson Efficacy Formula

Laplace-Smoothed Efficacy (No Division by Zero)

The old formula completion_rate × (1 / avg_attempts) divides by zero on new lessons and scores lessons with 1 completion perfectly (meaningless). v3.0 uses Laplace smoothing with a sample-size confidence discount:

α = 1, β = 2 # Laplace priors
smoothed = (completions + α) / (total_attempts + β)
n_threshold = 30
confidence = min(log(1 + n) / log(1 + n_threshold), 1.0)
efficacy = smoothed × confidence
difficulty = 1 - efficacy

# Gate: minimum 10 samples before flagging
flag if n ≥ 10 AND difficulty > 0.7 for 3+ days

New lesson with 0 attempts → efficacy = (0+1)/(0+2) × 0 = 0.5. Graceful default. Never NaN, never Infinity.

v3.0 Fix — Cold-Start Fallback Chain

Traffic Light Threshold — 3-Tier Resolution

The old spec said "if median < 10 samples → use lesson historical median" without defining what happens on day zero when the lesson is brand new. v3.0 defines a complete fallback chain:

  • TIER 1 Live session median (n ≥ 10 samples)
  • TIER 2 analytics_daily historical median (any n)
  • TIER 3 bootstrap_median_ms from lesson JSON

Bootstrap medians are seeded by the content team per lesson/step before pilot launch. Never null. Never GREEN by default. A day-zero classroom has real thresholds from minute one.

Threshold logic: GREEN < 2× median · YELLOW > 2× · RED > 3× OR 3+ fails

Observability, SLOs & Error Budgets

A system without SLOs is a system where you find out things are broken from teachers, not from your monitoring. This section defines what "working" means for Code4Kidz — per endpoint class, per role, with quantified error budgets and a tiered alert runbook.

FAANG Principle — "SLOs Before Features"

At Google, you don't ship a feature to production without defining its SLO. The SLO is the contract between the team and the users. The error budget is the finite resource that determines how much risk you can take. At Code4Kidz the users are 7-year-olds in a live classroom — a lesson endpoint going down mid-class is a political incident, not just a support ticket.

What to Monitor on Day One
  • Redis Streams consumer lag (XPENDING depth per consumer group) — if this grows, analytics events are queuing up
  • DLQ depth — if this is > 0, you're losing analytics data silently
  • WebSocket connection count vs expected (classroom size × active classrooms)
  • POST /progress p95 latency — this is the hot path, any spike is felt by students immediately
  • PostgreSQL query times on lesson_progress (watch for index misses as data grows)

The Golden Rules — Memorize These

Never log student data.

Pino structured logs must never include session slugs, classroom codes, or any field from student_sessions. The logger is configured with a redact list — but that's a safety net, not the plan. When in doubt, don't log it.

XP is always calculated server-side.

The client sends { lesson_id, step_id, idempotency_key }. The server computes XP from the lesson definition. A client claiming { xp: 500 } is ignored. Always.

Every external service needs a circuit breaker.

Wrap Redis calls in try/catch with a fallback. Wrap R2 signed URL generation with a cached fallback. The lesson must work even if half the infrastructure is down. A classroom session failing mid-class because Redis had a blip is a political problem.

All streak logic runs in org-local time, never UTC.

Look up the org's timezone field. Use date_trunc('day', ts AT TIME ZONE org.timezone) for all streak comparisons. A student doing their lesson at 11:58 PM should not lose their streak because your server clock thinks it's already tomorrow. This is the streak-loss bug that kills user retention in every edtech platform that doesn't think about timezones.

Redis Streams — XACK every processed event.

The consumer group tracks unacknowledged messages via XPENDING. If you process an event but forget to XACK it, it will be redelivered after the visibility timeout — you'll double-count analytics events. The /packages/streams wrapper handles this automatically: process your event inside the provided callback and it ACKs on success, routes to DLQ on failure after 3 retries.

Feature flags before feature branches go live.

New features ship behind a flag with rollout_pct: 0. We increase it to 10% (one classroom), then 50%, then 100%. Never all-or-nothing. This is how you catch a bug before it affects 1,800 schools — and how you protect the WebSocket error budget.
