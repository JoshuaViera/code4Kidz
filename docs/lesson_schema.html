Why This Schema Exists

The architecture doc defines how the system runs. The lesson doc defines what kids experience. This schema is the bridge. Without it, there's no contract between content authors and engineers — and the first time someone adds a lesson wrong, a classroom breaks mid-session.

v1.2 Changelog — April 2026

One new validation rule: all_instances_min_chars
Added for Lesson 3+ challenge steps where multiple required tags must all have meaningful content. content_min_chars only checks the first matching element — a student who writes one real paragraph and leaves the others empty would pass. all_instances_min_chars closes this gap by running querySelectorAll(tag) and checking every instance.

No new step types. No DB schema changes. No new error types. The new rule uses the existing empty_content error type.

One engineering task before CI will pass for lesson_003.json: Implement all_instances_min_chars in /packages/validators. Implementation: Array.from(doc.querySelectorAll(tag)).every(el => el.textContent.trim().replace(/\s+/g, '').length >= min).

FAANG Principle — "Content as Code"

At Duolingo, every exercise is a structured JSON object authored by linguists and validated by CI before it touches production. At Khan Academy, lesson content ships through a content pipeline with schema validation, not manual DB inserts. The pattern: content is data, data has a schema, schema is enforced at build time. A PR that adds a lesson with a missing bootstrap_median_ms fails the build. No exceptions.

Why JSON files — not a DB table?

Lessons are static, versioned artifacts. They don't change at runtime — they change through deliberate authoring. Storing them in the DB adds a CRUD API, a migration risk, and a content management UI you don't have time to build. JSON files in /data/lessons/ give you Git history, PR review, rollback, and diff for free. The DB tracks progress through lessons — not the lessons themselves.

Why is lesson_id a string like "lesson_001"?

Two reasons. First: human-readable and sortable. A PR titled "add lesson_006" is immediately obvious to every reviewer. Second: zero-padded integers sort correctly in both filesystems and API responses without a secondary order column. The number is the canonical curriculum order.

How does lesson_id connect to the DB?

The lesson_progress and mistake_log tables store lesson_id and step_id as plain strings — not foreign keys to a lessons table. The application validates that a submitted lesson_id matches a file in /data/lessons/ at startup. If the file doesn't exist, the server won't start. This is the contract.

Why is bootstrap_median_ms required on every step?

On Day 1 of a classroom session, analytics_daily has zero rows. Without a seed value, the traffic light system defaults to "unknown," which is useless. bootstrap_median_ms forces the content author to estimate how long a step takes. This is a pedagogical forcing function, not just a technical requirement.

The gap this closes in the architecture doc.

The architecture v3.1 references lesson_id and step_id in four tables: lesson_progress, mistake_log, analytics_daily, and feature_flags. It also requires bootstrap_median_ms as a required field in lesson JSON. But the architecture never defines what that JSON looks like. This document is that definition. Without it, every engineer who adds lesson 2 will invent their own format — and the validators will break.

Schema Reference

Every lesson JSON file must include these top-level fields. Required fields are marked REQUIRED. Optional fields are marked OPTIONAL.

Top-Level Structure

Field Type Status Description & Why
id string REQUIRED Format: lesson_001. Must match the filename. This is the primary key used in lesson_progress, mistake_log, and analytics_daily. Pattern: ^lesson_[0-9]{3}$. Start at 001, zero-padded to three digits.
schema_version string REQUIRED Semver string, e.g. "1.2.0". The validator checks this first. If you change the schema in a breaking way, bump the major version and update the validator. This prevents silent format drift.
version integer REQUIRED Content version of this specific lesson. Starts at 1. Increment when you edit lesson content after it has shipped to students. Used to detect stale cached content in Redis (TTL: 60s). Never reset to 1.
meta object REQUIRED Lesson-level metadata. See the Meta Object table below.
byte_arc object REQUIRED Byte's emotional journey across the lesson. opening_emotion and closing_emotion. Valid values: excited | neutral | curious | thinking | happy | proud | celebrating. The opening emotion sets Byte's state for the mascot intro step.
steps array REQUIRED Ordered array of step objects. Minimum 3 steps. Must include exactly one mascot_intro (order: 0 or 1 if warmup present) and exactly one lesson_complete (last). Steps are rendered in order field sequence, not array index.

The Meta Object

Field Type Status Description & Why
title string REQUIRED Display title shown to students. Max 40 chars.
slug string REQUIRED URL-safe string, e.g. say-hello-to-the-web. Used in API routes: GET /lessons/:slug. Pattern: ^[a-z0-9-]+$. Must be unique across all lessons.
emoji string REQUIRED Single emoji. Shown on the lesson card in the UI.
age_range object REQUIRED { "min": 7, "max": 10 }. All C4K lessons are 7–10 for now. This field exists so future curriculum tracks can be added without a schema change.
prior_lesson_id string|null REQUIRED null for lesson 1. Otherwise the id of the prerequisite lesson. The server checks lesson_progress to confirm completion before unlocking.
next_lesson_id string|null REQUIRED null for the last lesson. Otherwise the next lesson's id. Shown greyed out on the lesson complete screen as a teaser.
time_estimate_min integer REQUIRED Estimated lesson duration in minutes. Shown to facilitators. Must be honest — facilitators plan sessions around this.
total_xp integer REQUIRED Sum of all step XP values. Validated at build time: if the sum of steps[].xp doesn't equal this value, the build fails.
objective string REQUIRED One sentence. Pinned at the top of the screen. Max 60 chars. Written for a child. Not "Students will demonstrate..." — that's DoE spec language, not kid language.
concepts_introduced string[] REQUIRED Array of concept tags this lesson teaches. Used by analytics to track concept mastery across the cohort. Must come from the approved concept taxonomy.
concepts_required string[] OPTIONAL Concepts a student must have mastered from prior lessons. If a student has a high error rate on a required concept, the system surfaces a review prompt. Empty array for lesson 1.
tags string[] OPTIONAL Free-form tags for filtering in the teacher dashboard.

Fields Every Step Must Have

These fields are required on every step, regardless of type.

The step type then adds its own additional fields. See the Step Types tab for type-specific fields.

Field Type Status Description & Why
id string REQUIRED String ID for this step within the lesson. Pattern: ^[a-z0-9_]+$. Stored verbatim in lesson_progress.step_id and mistake_log.step_id. Never change a shipped step ID — it will orphan analytics data.
type string REQUIRED One of: warmup | mascot_intro | info | quiz | code_editor | challenge | lesson_complete | anatomy_builder. Determines which React component renders and which validation path runs.
order integer REQUIRED Zero-indexed render order. warmup is always 0 if present. mascot_intro is 0 (Lesson 1) or 1 (Lessons 2+). lesson_complete is always last. Frontend sorts by this field, not array index.
xp integer REQUIRED XP awarded when this step is completed. 0 for non-interactive steps. The server calculates XP from this field — the client never sends an XP amount, only a completion signal.
bootstrap_median_ms integer REQUIRED Estimated median time for this step in milliseconds. Required on every step including warmup and anatomy_builder. Used as tier-3 fallback when analytics_daily has fewer than 10 real samples. Time yourself doing it slowly, multiply by 1.5.
progress_label string|null REQUIRED String shown in the progress bar, e.g. "2 of 5". null for warmup, mascot_intro, challenge, anatomy_builder, and lesson_complete.
byte object REQUIRED { "emotion": "neutral", "lines": ["line 1", "line 2"] }. Max 2 lines. Valid emotions: excited | neutral | curious | thinking | happy | proud | celebrating.

Step Types

There are exactly 8 step types as of v1.2. Every lesson is composed of these building blocks. The type determines what fields are required beyond the base set, which React component renders, and whether the step writes to the DB.

DB write rules by step type.
  • warmup — NO writes to lesson_progress or mistake_log. Auto-advances. No server round-trip on failure.
  • anatomy_builder — NO writes to lesson_progress or mistake_log. Wrong drops are client-side only. Single completion signal when all labels correct.
  • mascot_intro, info — NO writes. Non-interactive.
  • quiz — Writes to mistake_log on wrong answer only.
  • code_editor — Writes to lesson_progress on completion. Writes to mistake_log on validation failure.
  • challenge — Writes to lesson_progress on completion.
  • lesson_complete — Writes streak + XP total to student_sessions. This is the only step that triggers the rollup.
warmupWarm-Up Review★ v1.1 · xp: 0 · order: 0 · NO DB write · Auto-advances at 60s
A 60-second review of the previous lesson's concept. No hearts at risk. No XP. No mistake_log write. Auto-advances when time_limit_ms elapses even if validation fails. First step of every lesson from Lesson 2 onward. Lesson 1 has no warm-up.
No server round-trip on failure.

The client handles the timeout locally and advances. The only server call is a completion signal if the student passes within the time limit — and even that is optional. Warm-up completion is NOT required to proceed.

The culminating moment. Byte explicitly steps back. No hints, no help. Always the second-to-last step. The skeleton code is given to prevent blank-page panic — the student fills in the content, not the structure.
Additional Field Type Description

anatomy_builderAnatomy Builder (Worked Example + Drag-Label)★ v1.1 · xp: 0 · NO DB write · Client-side on wrong drops
Two-phase step. Phase 1: Byte constructs a code structure token by token with narration and color-coded highlights. Phase 2: student drags sticky-note labels to the correct regions. Wrong drops bounce back client-side only. All labels correct → single server completion signal → advances.
DoE alignment: Worked Example Effect + Dual Coding.

Sweller (1988) — novice learners benefit from watching an expert complete a task before attempting it. Paivio (1991) — simultaneous verbal + visual encoding produces stronger retention. This is why the step has zero XP and zero wrong-answer penalty — it is orientation, not evaluation.

Validation System

Validation is the contract between what the lesson asks for and what the student submits. It runs in two places: on the client (immediate feedback) and on the server (source of truth). Both use the same validator package from /packages/validators. Never use regex.

FAANG Principle — "Why DOM validation, never regex?"

Regex-based HTML validation fails constantly in child-authored code. <H1> vs <h1>, trailing spaces, self-closing tags — children don't type predictably. A DOM parser handles all of these correctly. The validator parses the student's code into a real DOM, then queries it: doc.querySelectorAll('h1'). If it finds an h1, the tag is present. This is identical to how browsers validate HTML — which means if it passes the validator, it will actually render.

Validation Rules — 8 Total

Rule Fields What it checks Example
tag_present tag, error_type DOM contains at least one instance of the specified tag, properly opened and closed. 
{ "rule": "tag_present",
  "tag": "h1",
  "error_type": "missing_closing_tag" }
tag_count tag, min, error_type DOM contains at least min instances of the tag. Used where multiple tags are required. 
{ "rule": "tag_count",
  "tag": "p", "min": 2,
  "error_type": "wrong_tag" }
content_min_chars tag, min, error_type Text content inside the first matching tag (trimmed) must be at least min characters. Catches empty tags and whitespace-only content. 
{ "rule": "content_min_chars",
  "tag": "h1", "min": 2,
  "error_type": "empty_content" }
content_changed tag, not_equal, error_type Content inside the tag must differ from the starter value. Used in personalization steps — ensures the student changed the placeholder. 
{ "rule": "content_changed",
  "tag": "h1",
  "not_equal": "Hello, I'm Byte!",
  "error_type": "empty_content" }
all_tags_valid allowed_tags, error_type Every tag in the student's code must be in the allowed_tags list. Prevents students from using tags they haven't learned yet. 
{ "rule": "all_tags_valid",
  "allowed_tags": ["h1","p"],
  "error_type": "wrong_tag" }
css_property_set v1.1 selector, property, not_default, error_type Checks getComputedStyle(doc.querySelector(selector))[property] against the browser default. Passes if the value differs. Default for color: rgb(0,0,0). Default for background-color: rgba(0,0,0,0). Always use computed style — never string-match raw CSS. 
{ "rule": "css_property_set",
  "selector": "h1",
  "property": "color",
  "not_default": true,
  "error_type": "css_property_error" }
css_has_semicolon v1.1 selector, property, error_type Parses the style tag's text content and checks that the rule targeting selector for property ends with a semicolon. Catches syntactically incorrect CSS regardless of browser forgiveness. 
{ "rule": "css_has_semicolon",
  "selector": "h1",
  "property": "color",
  "error_type": "css_syntax_error" }
all_instances_min_chars v1.2 tag, min, error_type Queries ALL instances of the tag using querySelectorAll(tag) and checks that every one has textContent.trim().replace(/\s+/g,'').length >= min. Fails on the first instance that doesn't meet the minimum. Why this exists: content_min_chars only checks the first matching element — a student who fills one <p> and leaves the others empty would pass. This rule closes that gap. Required for Lesson 3+ challenge steps with multiple required content tags. Implementation: Array.from(doc.querySelectorAll(tag)).every(el => el.textContent.trim().replace(/\s+/g,'').length >= min). 
{ "rule": "all_instances_min_chars",
  "tag": "p",
  "min": 5,
  "error_type": "empty_content" }

Error Types — The Enum (7 Values)

The error_type field on each validation rule maps directly to the error_type enum in the mistake_log DB table. This is the link between lesson content and learning analytics.

error_type value What happened What the teacher dashboard surfaces
missing_closing_tag Student wrote <h1>Hello without </h1>. Or wrote nothing at all. "X% of kids on step Y are missing the closing tag" → Instructor review flag
wrong_tag Student used a tag not in the allowed set, or the wrong tag for the task. "Top error on step Y is wrong_tag" → Lesson may need clearer instruction
empty_content Tag is present but contains no content, whitespace only, or fewer than the minimum characters. Also fired by all_instances_min_chars when any instance is empty. "High empty_content rate on step Y" → Hint delay may need to decrease
case_error Reserved: student used <H1> instead of <h1>. DOM validator passes this (HTML is case-insensitive) but flagged for style feedback in advanced lessons. Not currently surfaced. Reserved for Lesson 6+.
whitespace_only Tag content is all whitespace. Distinct from empty because the student tried to enter something. "Whitespace_only on step Y" → Student may be pressing space bar testing the editor
css_syntax_error v1.1 Student wrote a CSS rule missing a semicolon or colon. Fired by css_has_semicolon. "css_syntax_error on step Y" → Lesson may need to emphasize semicolons more explicitly
css_property_error v1.1 Student left a CSS property value empty or used a value that doesn't change computed style from default. Fired by css_property_set. "css_property_error on step Y" → Hint delay may need to decrease
How Byte's wrong-answer behavior maps to error types.

Byte never says "That's wrong." Byte redirects based on what went wrong. The error_type from the validator determines which Byte line plays. missing_closing_tag → "Find the one with the little line inside." empty_content → "Click inside. Write something." The lesson author writes these lines per-step in the feedback.incorrect field.

Storage Guide — Where Everything Lives

Match the tool to the job. The wrong answer: one big Google Doc with everything. The right answer: the source of truth for a given artifact type should be the tool that makes it hardest to let that artifact go stale.

FAANG Principle — "Single Source of Truth per Artifact Type"

At Stripe, API documentation lives in the codebase as OpenAPI specs — not in Notion. At Airbnb, design tokens live in Figma and are programmatically exported to code. The principle: the source of truth for a given artifact type should be the tool that makes it hardest to let that artifact go stale. For code and JSON: Git. For narrative content and planning: Notion.

Adding a New Lesson — Checklist

This is the process every new lesson must go through. Print this. Follow it. If any step is skipped, the lesson doesn't ship.

Phase 1 — Content (Notion)

  • Write the narrative spec — Full lesson in Notion following the lesson spec format. Title, objective, Byte dialogue, step-by-step content, writing rules applied throughout. Gets a Notion status: "Spec."
  • Apply the writing rules — Max 8 words per sentence. No word above 2 syllables unless it's the concept being taught. Byte speaks max 2 lines. Wrong answers redirect, never punish.
  • Define the concept arc — What exactly does this lesson teach? Update concepts_introduced. Confirm it builds on the prior lesson's concepts_introduced.
  • Estimate step timings — For each step, estimate bootstrap_median_ms. Time yourself doing it slowly. Multiply by 1.5 for a 7-year-old.
  • DoE + peer review the spec — At least one other person reads it. Apply the DoE inspector checklist from the lesson spec's DoE section. Would a 7-year-old understand this?

Phase 2 — JSON (GitHub PR)

  • Create the file — /data/lessons/lesson_00N.json. Copy the structure from lesson_003.json as the current best-practice starting point.
  • Set schema_version to "1.2.0" — All lessons from Lesson 3 onward use schema v1.2. Do not use "1.0.0" or "1.1.0" — the CI validator will reject it.
  • Fill in meta — Set id, schema_version, version: 1, all meta fields. Confirm prior_lesson_id and next_lesson_id. Set total_xp.
  • Translate each step — Every step in the narrative spec becomes a step object. Assign step IDs following the pattern: step_1_info, step_1_quiz, step_2_editor. IDs never change after first ship.
  • Set bootstrap_median_ms on every step — No step can have this field missing. The CI validator will reject the file.
  • Use all_instances_min_chars for challenge p tags — Any challenge step with multiple required content tags must use all_instances_min_chars, not just content_min_chars. Schema v1.2 requirement.
  • Verify XP sum — Sum of all step XP values must equal meta.total_xp. The validator checks this.
  • Run the validator locally — npm run validate:lessons. Fix all errors before opening a PR.

Phase 3 — Review & Ship

  • Open the PR — Title: content: add lesson_00N — [Lesson Title]. Link to the Notion spec page in the PR description.
  • CI passes — Schema validation, XP sum check, step ID uniqueness, bootstrap_median_ms presence, all_instances_min_chars usage check. All green.
  • Manual playthrough — Someone on the team runs through the full lesson on the preview deploy. Every step, including wrong answers and the challenge.
  • Update Notion status — Change the lesson's status from "Spec" → "Shipped."
  • Update prior lesson's next_lesson_id — If lesson 4 ships, lesson 3's next_lesson_id needs updating. This unlocks the teaser on the lesson complete screen.
Never change a step ID after it has shipped to students.

Changing a step ID will orphan all the lesson_progress and mistake_log rows for that step. The analytics dashboard will silently lose that data — and students will be asked to redo a step they've already completed. If you need to restructure steps, create new step IDs and migrate the old data. This is a two-phase operation documented in MIGRATION_GUIDE.md.
